## Сверка с эталоном (обязательна перед UI)
Папка `engine_reference/` — работающий код на `pyswisseph/pysweph`. Любой новый
движок обязан давать ИДЕНТИЧНЫЙ результат: долготы до 0.01°, времена точных аспектов
до минуты.

**РЕЗУЛЬТАТ (2026-06-27).** Спайк в `spike/`. `dump_reference.py` дампит эталон
(pysweph 2.10.03, фактически Moshier — файлов эфемерид нет) в `reference_dump.json`
(позиции, аспекты, затмения+Сарос). Три verify-скрипта гоняют **swisseph-wasm**
(Moshier) через общий `engine.mjs` и сверяют: `verify_wasm.mjs` (позиции),
`verify_aspects.mjs` (времена аспектов), `verify_eclipses.mjs` (затмения §10.1).
Даты позиций: 2000-01-01, 2010-09-09, 2020-12-21, 2023-03-02, 2025-01-01, 2026-06-27.
Итог: позиции до **0.0000″**, аспекты ≤ **0.49 сек**, Сарос **10/10**. ✅

Прогон: `cd spike && npm install && npm run verify`. Пересборка эталона —
`../.venv/Scripts/python.exe dump_reference.py` (через `npm run dump:ref` путь к
python в cmd не резолвится — звать python напрямую).

**Попутно исправлен баг ЭТАЛОНА** (`engine_reference/daily_aspects_astro.py`,
`_exact_time`): точное время не считалось для СОЕДИНЕНИЙ и ОППОЗИЦИЙ (там
`|sep|-target` лишь касается 0/180 и не меняет знак → было `exact_time=None` для всех
☌ и ☍). Теперь для ☌ ищем ноль знаковой `angdiff(lon1,lon2)`, для ☍ — `angdiff(delta,
180)`. Проверено: Юпитер–Сатурн 2020-12-21 → 18:20:37 UTC (=0°29′10″ Водолея).
Оставшиеся `None` у ☌ — легитимные «близкие прохождения» (мин. разница >0, точного 0°
нет, напр. Нептун–Кету 0.25°).

**Окно орбиса (требование астролога).** Аспект — ИНТЕРВАЛ, не «весь день». В записи
аспекта `begin_time`/`end_time` (вход/выход орбиса) рядом с `exact_time`
(`_orb_distance`/`_orb_edge` в эталоне). Окно может выходить за сутки: Юпитер–Сатурн ☌
2020-12 — окно 12→30 дек (18 дней); Луна–Сатурн ⚹ — вход 21 дек 21:32, выход 22 дек
01:33 (через полночь).

## Выбор пакета WASM (2026-06-27): swisseph-wasm, не @swisseph/browser
Взят **swisseph-wasm** (prolaxu), а НЕ @swisseph/browser@1.1.1. Причина: @swisseph
отдаёт у затмений только времена (нет `*_eclipse_how`/`*_where` в экспортах) → §10.1
(Сарос) невыполним; плюс dist-бандл сломан (CJS-куски в ESM, ссылки свёрнуты в
`(void 0)` — жил на заплатке patch_vendor.mjs). swisseph-wasm — штатный emscripten-glue
(заплаток НЕ нужно), экспортирует ПОЛНЫЙ низкоуровневый API (sol/lun eclipse
when/where/how) → Сарос берём через `Module.ccall` (см. `spike/engine.mjs`). Бонус: в
пакете есть полные файлы эфемерид (`swisseph.data`, ~12 МБ). Минусы: пакет молодой
(v0.0.5, один автор) и GPL-3 — но это лишь сборка канонического C-исходника Swiss Eph.
Сарос: серия=`attr[9]`, член=`attr[10]` из `*_eclipse_how`; для солнечных сперва
`sol_eclipse_where` (точка максимума), иначе how вернёт 0.

## Доп. объекты и догрузка (астероиды/TNO) — проверено 2026-06-27
Режим SWIEPH. Код объекта: астероид = `SE_AST_OFFSET(10000) + номер`. Расчёт — обычный
`swe_calc_ut(jd, ipl, FLG_SWIEPH|FLG_SPEED)`.
- **В бандле `swisseph.data`** (смонтирован в `/sweph`): `sepl_18`, `semo_18`,
  `seas_18` + таблицы. Считаются СРАЗУ: Хирон(15), Фолус(16), Церера(17), Паллада(18),
  Юнона(19), Веста(20), Лилит ср.(12)/ист.(13). Спайк: ✅ все.
- **Чего нет в бандле:** индивидуальные файлы TNO. Имя файла даёт сама библиотека в
  тексте ошибки: Эрида → `s136199.se1` (длинный, 3000 BCE–2999 CE) или `s136199s.se1`
  (короткий 1500–2100); номера ≥100000 начинаются с `s`, <100000 — с `se`. Файл ОБЪЕКТА
  кладётся в подпапку `astN` (N=номер/1000): Эрида → `/sweph/ast136/s136199.se1`.
- **Догрузка во время работы РАБОТАЕТ** (проверено): у emscripten-модуля доступна
  виртуальная ФС — `swe.SweModule.FS.writeFile('/sweph/astN/<файл>.se1', bytes)`, затем
  `calc_ut` сразу видит объект. Спайк `spike/probe_extra_objects.mjs`: Церера на 1000 г.
  падала «нет `seas_06.se1`» → после записи файла в ФС посчиталась.
- **РЕШЕНО: набор астролога ВШИТ в APK** (оффлайн-первый; ~15–20 КБ на объект, Эрида
  171 КБ — итого ~0.3 МБ). Астролог прислала 19 файлов (папка `sw/`), разложены в
  `ephe/astN/`: Хаос(19521), Варуна(20000), Иксион(28978), Хуйя(38628), Квавар(50000),
  2002 UX25(55637), 2003 VS2(84922), Седна(90377), Оркус(90482), 2004 GV9(90568),
  Салация(120347), Хаумеа(136108), Эрида(136199), Макемаке(136472), Варда(174567),
  2003 AZ84(208996), Гонгун(225088), 2002 MS4(307261), Лелеакухонуа(541132). Все, кроме
  Эриды — «короткие» (1500–2100). При сборке копируются в ассеты, на старте пишутся в
  `Module.FS`. Имена/коды — data-driven (скан `ephe/ast*/`), см. `dump_swieph.py`.
- **Источник файлов TNO:** Dropbox Алоиса (`long_ast/astN/` длинные, `all_ast/astN/`
  короткие) — но это JS-страница, СКРИПТОМ (curl) НЕ качается (отдаёт HTML-обёртку;
  настоящий файл начинается со слова `SWISSEPH`). Файлы взяты ГОТОВЫЕ у астролога. Ядро
  (`sepl_18/semo_18/seas_18`) — на GitHub `aloistr/swisseph/ephe` (в `ephe/` для
  питон-эталона; в APK не нужно — внутри `swisseph.data`).

**СВЕРКА SWIEPH↔SWIEPH (2026-06-27).** `spike/dump_swieph.py` (pysweph с файлами из
`ephe/`) ↔ `spike/verify_swieph.mjs` (swisseph-wasm; те же файлы пишутся в его ФС →
читают идентичные данные). **38 объектов** × 6 дат = **228 значений**, 0 ошибок: макс.
ошибка долготы **0.069″** (≈2·10⁻⁵°, худший — Марс; TNO ещё точнее), скорости
4·10⁻⁷°/сут — внутри допуска с запасом. ✅ Прогон: `cd spike &&
../.venv/Scripts/python.exe dump_swieph.py && npm run verify:swieph`.

Пример «почему числа зависят от времени»: Великое соединение Юпитер–Сатурн 2020-12-21.
На 12:00 UTC Сатурн 0°27′ Водолея, но ТОЧНОЕ соединение — в 18:20:36 UTC, и тогда обе
планеты ровно в **0°29′10″ Водолея**. Снимок на полдень ≠ момент точного аспекта.

## Доп. пожелания астролога — раунд 3 (2026-06-27)
Поверх ЗАДАНИЯ §3.5–3.7:
- **Архетипы божеств греческой мифологии — на КАЖДУЮ планету** (не только на сигнатуру
  аспекта). У объекта — редактируемый текст-архетип бога. Связать с §3.7.
- **Бортовой журнал = заметки/наблюдения** (§3.5). Из записи — ИИ-анализ «на совпадение
  с мифом» (модель сопоставляет наблюдение с архетипом). ИИ только по запросу, на ключе.
- **Виды журнала:** группировка/фильтр по периоду (день/неделя/месяц) + выборка по
  конкретной планете ⇒ запись тегируется датой и объектом(ами).
- **Функция СРАВНЕНИЯ заметок:** выбрать 2+ записи и сравнить.
- **Ввод своего токена Claude** — ключ только в защищённом хранилище. Перед ИИ-частью
  свериться со справкой по Claude API (skill `claude-api`).

## Раунд 4 — отзывы астролога (2026-06-30, реализовано)
Веб-часть, проверено `npm run check` + `node test/parity.mjs` + `npm run build`:
- **Заметки «не сохранялись»** — баг реактивности Svelte 5: db-коллекции возвращают ТУ
  ЖЕ ссылку массива, `put`/`remove` мутируют на месте → присваивание не триггерит
  `$state`/`$derived`. ПРАВИЛО: в компонентах брать `db.x.all().slice()`.
- **Единый часовой пояс:** сутки/аспекты/события — на 00:00 ВЫБРАННОГО пояса
  (`zonedDayStartUTC`/`todayCivil`/`fmtDayMid` в `lib/format.ts`).
- **Колесо/положения — на «настоящий момент»** (тик раз в 60с, `DayScreen` `snapshot`);
  аспекты — за все сутки (интервалы).
- **Порядок аспектов** по удалённости от Солнца (`SUN_RANK`/`sunRank`, `constants.ts`);
  в карточке — ДАТЫ входа/выхода.
- **Орбис по КАЖДОМУ объекту** (`Settings.orbs`), для пары БОЛЬШИЙ из двух; резолвер
  `orbResolver` (`models.ts`); `aspectsOn` принимает число ИЛИ функцию-резолвер.
- **Нижняя панель** (Дата/Журнал/Библиотека/Чат/Настройки); **Библиотека** вынесена в
  `LibrarySheet` + `InterpretationsSheet`.
- **Свайп-вниз закрывает шторку** + блок скролла фона — общий экшен `lib/sheet.ts`.
- **Настройки сгруппированы**; «Крупный шрифт» (`Settings.largeFont`).
- **Геолокации НЕТ** (деанона нет): пояс из `Intl...timeZone`. Форум — НЕ делать.
- **OTA — Capgo** (`@capgo/capacitor-updater`, ручной режим `autoUpdate:false`).
  `lib/ota.ts`: старт → `notifyAppReady()` → fetch `apk/latest.json` → если версия
  новее → `download` + `next`. CI зипует `dist`→`apk/bundle.zip` + пишет `latest.json`.

## Раунд 6 — доделки + фикс сохранения (2026-07-01)
Проверено `npm run check` (0/0). ⚠️ keystore/`build.gradle` — НАТИВНЫЕ правки, нужна
свежая сборка APK через CI; веб-правки едут по OTA.
- **✅ Фиксированный keystore.** `signingConfigs.debug` в `app/android/app/build.gradle`
  → закоммиченный `astra-debug.keystore`; CI (`build-apk.yml`) генерит его через
  `keytool` один раз и коммитит (`git add -f`), дальше переиспользует → подпись APK
  стабильна → обновление ставится ПОВЕРХ без потери данных. `.gitignore` — точечное
  исключение из `*.keystore`. ⚠️ РАЗОВО при переходе на новый ключ данные слетят
  (подпись сменилась с прежней рандомной): экспорт → переустановка → импорт, дальше без
  потерь.
- **✅ Настоящий баг «данные не сохраняются»** (`lib/db.ts`): `serialize()` пишет ОБЁРТКУ
  `{app,schema,savedAt,data:{…}}`, а `loadLS()` читал её как ПЛОСКУЮ → коллекции
  поднимались ПУСТЫМИ. Ввёл `payload()` (достаёт `.data`), применил в `loadLS` и `adopt`.
- **✅ Автосохранение переписки с Claude** (`ui/ChatSheet.svelte`): после каждого ответа
  разговор пишется в журнал сам — ОДНА запись на разговор (upsert по `noteId`). Из
  карточки аспекта (`App.openChat(seed, source)`) — тег объектов+сигнатуры. Перенос на
  другой телефон — штатным экспортом/импортом; P2P не делаем.
- **✅ Свайп-закрытие шторок не такое резкое** (`lib/sheet.ts`): «потянуть-закрыть»
  смотрит на ближайший ВНУТРЕННИЙ скролл под пальцем; порог 90→130 px.
