# План: сообщество астрологов на Supabase (обсуждения аспектов, лайки, комментарии)

Статус: ПЛАН (2026-07-02), не реализовано. Отвечает на идею из «правки 2806» п.9
(«форум астрологов для своей тусовочки») + запрос владелицы: быстрая регистрация
через Google, обсуждения аспектов с лайками и комментариями.

## Главный принцип — не ломаем оффлайн-первое

Ядро Astra (небо, журнал, библиотека) остаётся полностью оффлайн. Сервер — это
ОПЦИОНАЛЬНЫЙ слой «Сообщество»: без сети и без входа всё работает как сейчас.
Вход через Google нужен только чтобы читать/писать обсуждения.

## Почему Supabase

- Postgres + Auth (Google OAuth из коробки) + Row Level Security + Realtime —
  весь бэкенд без своего сервера.
- Бесплатный тариф: 500 МБ БД, 50k активных пользователей/мес — для «своей
  тусовочки» хватит с запасом (вместо оценённых 400 ₽/мес за VPS — 0 ₽).
- Один публичный anon-ключ в клиенте — это норма: данные защищает RLS, не ключ.

Уточнение терминов: «задеплоить приложение НА Supabase» нельзя — Supabase не
хостит статические сайты. Supabase = БД/вход/API. Веб-версию приложения
(dist) раздаём через **GitHub Pages** (репо и CI уже есть, бесплатно). APK
живёт как раньше; веб-адрес станет и «форумом».

## Схема данных (SQL-миграция)

```sql
profiles       id uuid PK = auth.uid, display_name text, avatar_url text, created_at
discussions    id uuid PK, aspect_signature text NOT NULL,  -- lib/signature.ts, уже есть!
               exact_at timestamptz,       -- момент точного аспекта (группировка по событию)
               title text, body text, author_id -> profiles, created_at
comments       id uuid PK, discussion_id -> discussions, author_id -> profiles,
               body text, created_at
likes          user_id -> profiles, target_kind text CHECK (in ('discussion','comment')),
               target_id uuid, PRIMARY KEY (user_id, target_kind, target_id)
```

Индексы: `discussions(aspect_signature, created_at desc)`, `comments(discussion_id, created_at)`.

RLS-политики:
- SELECT — всем вошедшим (решить: или вообще публично на чтение);
- INSERT — только со своим `author_id = auth.uid()`;
- UPDATE/DELETE — только автор своей записи;
- лайк уникален по PK, снять можно только свой.

Счётчик лайков — `count(*)` в запросе (для маленького сообщества достаточно,
триггерные счётчики — потом, если вырастет).

## Ключевая склейка с приложением

`aspect_signature` из `lib/signature.ts` — естественный ключ ветки обсуждений.
В шторке аспекта (InterpretationSheet) появляется блок «Обсуждения» — треды по
этой сигнатуре. Из ленты сообщества тап по обсуждению открывает день аспекта
(через уже существующий `ongoto`). То есть форум «пришит» к небу, а не живёт отдельно.

## Вход через Google (Capacitor)

- Supabase Auth → провайдер Google. В вебе — обычный redirect.
- В APK — `signInWithOAuth` через системный браузер (@capacitor/browser) +
  deep link возврата `app.astra.sky://auth-callback` (intent-filter в манифесте —
  потребуется НОВЫЙ APK, OTA нативку не меняет).
- Сессию Supabase SDK хранить через адаптер на `@capacitor/preferences`
  (localStorage на Android не durable — известные грабли).

## Этапы работ

1. **Владелица**: завести проект на supabase.com (вход через GitHub) + в Google
   Cloud Console создать OAuth client (дам пошаговую инструкцию) → два ключа
   (URL проекта + anon key) кладём в конфиг приложения (не секрет).
2. SQL-миграция (таблицы+RLS выше) через Supabase SQL editor.
3. Клиент: `@supabase/supabase-js`, новый модуль `lib/community.ts`
   (вход/выход, треды по сигнатуре, комменты, лайки). UI её не ждёт: без сети
   блок «Обсуждения» просто показывает «офлайн».
4. UI: блок в InterpretationSheet + вкладка «Сообщество» (лента свежих
   обсуждений). Лайк — сердечко с оптимистичным обновлением.
5. Деплой веб-версии: GitHub Pages workflow (build dist → Pages). Домен
   `gggssaggg-beep.github.io/astra`.
6. Новый APK (deep link для входа) + OTA для веб-части.

Оценка: 2–3 рабочие сессии. Риски: настройка Google OAuth (одноразовая возня),
модерация (пока — автор удаляет своё; роль «админ» добавить позже).

## Решения, которые нужны от владелицы

1. Чтение обсуждений: только для вошедших или публично?
2. Имя в сообществе: из Google-профиля или свой ник?
3. Веб-версия становится публичной (GitHub Pages) — ок? (данные журнала всё
   равно локальные у каждого, на сервер уходят только обсуждения).
