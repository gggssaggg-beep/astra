# ЗАДАНИЕ для Claude Code — локальное Android-приложение для астролога

> Рабочее имя: **Astra** (поменяй, если нужно).
> Формат: это ТЗ, а не код. Код пишет Claude Code в терминале.
> Язык интерфейса: русский. Платформа: Android, оффлайн-первый.

---

## 0. Принципы (читать до начала работы)

1. **Сперва ищем готовые решения.** Для эфемерид, отрисовки колеса карты,
   свайпов, календаря, уведомлений, хранения — брать существующие
   библиотеки/плагины, а не писать с нуля. Любой «велосипед» обосновывать.
2. **Минимум кода, токен-экономия.** Не плодить фреймворки и тяжёлые
   зависимости. Расчёты — детерминированно, без ИИ. ИИ подключается ТОЛЬКО
   в чате трактовок по явному запросу (раздел 3.6).
3. **Оффлайн-первый.** Всё ядро (расчёты, лента дней, колесо, заметки)
   работает без интернета. Сеть нужна только для чата трактовок.
4. **UI — на готовых компонентах.** UX-фичи продумать заранее (раздел 3, 5).
   Визуальное направление для Claude Design — раздел 5.
5. **Жёсткие правила расчёта — НЕ нарушать** (раздел 2). Это перенос правил
   из уже работающего бота. Любой движок обязан давать результат, идентичный
   питон-эталону из `engine_reference/` (сверка до 0.01°).

## 1. Архитектура: две развилки, решить ДО строительства UI

UI делается **движок-независимым**: весь расчёт спрятан за интерфейсом
(набор функций, раздел 2). Тогда выбор движка не влияет на экраны, и работу UI
нельзя «потерять» при смене движка.

### Развилка A — платформа (рекомендация: web + Capacitor → APK)

- **Рекомендуется:** веб-приложение (ванильный JS или лёгкий слой) +
  **Capacitor** для упаковки в APK. Почему: переиспользует имеющийся опыт PWA,
  токен-экономно, а плагины Capacitor дают нативные **уведомления, хаптику,
  deep-link, защищённое хранилище** — ровно то, что просят пункты 1, 3, 4.
  Астролог ставит APK, всё работает оффлайн.
- **Не рекомендуется:** Flutter / React Native — тяжелее, противоречит принципу
  «минимум кода».

### Развилка B — движок эфемерид на устройстве (КРИТИЧНО)

Питон+C (`pyswisseph`/`pysweph`) на Android в вебе напрямую не работает.
Жёсткое правило проекта — «только Swiss Ephemeris». Поэтому:

- **Вариант B1 (рекомендуемый для веб-пути):** Swiss Ephemeris, собранный в
  **WebAssembly**, в режиме **Moshier** (не требует файлов эфемерид). Полностью
  оффлайн в веб-слое, математика Swiss Eph сохранена. Найти поддерживаемый
  WASM-билд (например, пакеты вида `swisseph-wasm` / `sweph` — проверить
  актуальность и лицензию).
- **Вариант B2 (питон-точный):** нативный Android через **Chaquopy**, внутри —
  `pysweph`, а UI тот же веб в WebView. Сохраняет ровно `pysweph`, но это уже
  нативная сборка (Android Studio, Kotlin) — тяжелее.
- **Вариант B3 (НЕ по умолчанию):** чисто-JS библиотека (например
  `astronomy-engine`). Самый лёгкий, оффлайн, но это **НЕ Swiss Ephemeris** →
  нарушает жёсткое правило. Использовать ТОЛЬКО с явного письменного согласия
  владелицы/астролога.

### Первый шаг проекта — «спайк движка» (go/no-go)

Прежде чем рисовать экраны:
1. Поднять кандидат-движок (B1, при провале — B2).
2. Прогнать его на контрольных датах и **сверить с питон-эталоном**
   (`engine_reference/`, см. раздел 9). Совпадение долгот планет, аспектов,
   границ орбиса — **до 0.01°** и до минуты по времени точных аспектов.
3. Зафиксировать выбор B1/B2 в `CLAUDE.md`. Только после этого — UI.

## 2. Движок расчётов (ядро)

### 2.1 Жёсткие правила (перенести из бота, НЕ менять)

- Только Swiss Ephemeris (через WASM или Chaquopy). **Никаких самодельных
  формул эфемерид.**
- **Тропический** зодиак (никакой сидерики/Джйотиш).
- Все расчёты во **времени UTC**; перевод в часовой пояс — только при выводе.
- **Аудит элонгаций:** Меркурий <~28° от Солнца, Венера <~48°. Иначе —
  пометка «расчёт сломан».
- Аспект подтверждается **математической разницей градусов**, а не картинкой.
- Объекты по умолчанию: Солнце, Луна, Меркурий, Венера, Марс, Юпитер, Сатурн,
  Уран, Нептун, **истинные** Лунные узлы (Раху/Кету). Состав — настраиваемый.
- Мажорные аспекты: соединение ☌, секстиль ⚹ (60°), квадрат □ (90°),
  трин △ (120°), оппозиция ☍ (180°). Орбис — настраиваемый, по умолчанию 1°.

### 2.2 Что портировать из `engine_reference/`

Логика уже написана и проверена на `pyswisseph`. Перенести 1-в-1 (см. файлы):

- Список объектов `BODIES`, словарь `ASPECTS`, глифы — из
  `daily_aspects_astro.py`.
- **Аспекты за день** с точным временем кульминации и пометкой
  applying/separating (→/←) — `aspects_for_day()` там же.
- **Окно аспекта** (вход в орбис → точный момент → выход), включая
  обработку ретроградного «качания» (несколько входов) — `_orb_window()` и
  `transit_conjunctions()` в `transits_astro.py`.
- Аудит элонгаций — `physics_audit()`.

### 2.3 Интерфейс движка (что UI ждёт от ядра)

Спроектировать тонкий слой с примерно такими функциями (имена свободные):

- `positions(at_utc)` → долготы/знаки/скорости/ретро всех объектов.
- `aspects_on(day, orb, include_moon)` → список аспектов дня
  (участники, тип, орбис, точное время, applying/separating).
- `aspect_window(p1, p2, aspect, around_date, orb)` → «эпизод» аспекта:
  время входа в орбис, время(на) точного, время выхода. Для медленных планет
  эпизод может тянуться днями/неделями и иметь несколько точных моментов
  (ретроградность) — отдать все.
- `find_aspect(p1, p2, aspect, date_from, date_to, orb)` → все эпизоды в
  диапазоне (для поиска, раздел 3.3).
- `events_on(day)` → «события неба» помимо аспектов: смена знака планетой
  (ингрессия), станции (переход ретро↔директ), новолуние/полнолуние,
  затмения. Это то, что астрологу нужно видеть отдельно (раздел 3.3).

Всё детерминированно, без сети и ИИ.

## 3. Экраны и UX (по пунктам владелицы)

### 3.1 Лента дней свайпом + календарь (п.1)

- Горизонтальный свайп влево/вправо = предыдущий/следующий день, с **хаптиком**
  (Capacitor Haptics) на смене дня. Плавная пружинная анимация, лёгкий параллакс
  фонового звёздного неба.
- По запросу — **выпадающий календарь** (нижний лист/bottom sheet) для прыжка на
  дату. Плюс быстрый ввод даты строкой (см. 3.3).
- Экран дня: дата, мини-«фото неба» (3.8), список аспектов карточками,
  блок событий дня (ингрессии/станции/лунации), блок Луны отдельно.

### 3.2 Карточка аспекта: орбис и тайминги (п.2.1)

Каждый аспект — карточка с: глифами участников, типом, текущим орбисом,
стрелкой →/← (приближается/расходится) и **временами**: вход в орбис →
точный момент → выход из орбиса (все в часовом поясе пользователя).
Орбис **настраиваемый** (глобально и, желательно, на конкретный аспект).
Изменение орбиса мгновенно пересчитывает границы окна.

### 3.3 Отслеживаемые аспекты, их «трансформация» и поиск в диапазоне (п.2)

- **Закрепление (pin/favourite):** отметить интересующие пары+аспект; они
  выносятся в отдельную панель «Отслеживаю».
- **Трансформация во времени:** для закреплённого аспекта — мини-таймлайн:
  когда входит в орбис, когда точный, когда выходит; орбис «сжимается»
  визуально по мере приближения к точному.
- **Что астрологу обычно нужно отдельно** (предусмотреть как готовые фильтры):
  точные моменты аспектов; **ингрессии** (планета меняет знак); **станции**
  (ретро/директ); **лунации** (ново/полнолуния) и **затмения**; **возвращения**
  планет. Эти события подсвечивать и давать на них тоже заметки/напоминания.
- **Поиск аспекта в диапазоне:** «найти Сатурн □ Нептун между A и B». Диапазон
  задаётся **вводом дат строкой** (парсить свободный ввод: `12.03.2026`,
  `2026-03-12` и т.п.). Результат — список эпизодов с таймингами, тап → карточка.

### 3.4 Уведомления (п.3) + tap открывает конкретное событие (п.4)

- **Ежедневное** уведомление в заданное время (по поясу пользователя) — краткая
  сводка дня.
- **На каждый аспект — настраиваемо:** «напомнить за X дней» до точного момента,
  ИЛИ хаптик/уведомление **в момент** точного аспекта.
- Реализация «в момент»: заранее вычислять точные времена и **планировать
  локальные уведомления** на эти метки (Capacitor LocalNotifications). Фон
  не нужен — ОС сама поднимет в нужное время.
- **Tap по уведомлению в шторке → открывается именно это событие** (карточка
  аспекта/дня), а не главный экран. Через `extra`-payload уведомления +
  роутинг к экрану события (deep-link внутри приложения).

### 3.5 Заметки дня и аспекта + «похожие прошлые» (п.5)

- Заметка на **день** и заметка на **конкретный аспект** (как проявилось).
- У каждого аспекта есть **сигнатура** (планета1+планета2+тип). «Показать
  похожие прошлые» = выбрать все прошлые заметки с той же сигнатурой → астролог
  видит, как этот аспект проявлялся в прошлые разы. Очень ценно для отслеживания
  паттернов.
- Поиск по заметкам, сортировка по дате/сигнатуре.

### 3.6 Чат трактовок на своём API-ключе (п.6)

- Поле для ввода **своего ключа Anthropic** (хранить в защищённом хранилище
  устройства, не в коде, не в гите).
- Чат, где можно обсудить трактовку текущего аспекта/расклада дня. В запрос
  модели вкладывать **уже посчитанные данные** (позиции, аспекты, тайминги) —
  модель их трактует, не пересчитывает.
- **CORS:** вызывать API через нативный HTTP Capacitor (CapacitorHttp), чтобы
  обойти ограничения браузера; либо заголовок прямого браузерного доступа.
  Ключ — пользователя, на его устройстве.
- Без ключа приложение полностью работает (чат просто выключен).

### 3.7 Библиотека трактовок — греческая мифология (п.7)

- Локальная **редактируемая библиотека трактовок**, ключ = сигнатура аспекта
  (и опц. знак/дом). Астролог пишет/правит свои тексты со ссылками на греческую
  мифологию; приложение их хранит и показывает рядом с аспектом.
- Из чата (3.6) можно «сохранить ответ в библиотеку» как черновик трактовки.
- Трактовки — её, приложение их запоминает и переиспользует.

### 3.8 «Фото неба» без токенов (п.8)

**Да, без токенов и без ИИ.** Это детерминированная **векторная отрисовка
колеса** (SVG/Canvas) по посчитанным позициям: круг зодиака, планеты на своих
градусах, линии аспектов. Брать готовую библиотеку колеса карты (например
`@astrodraw/astrochart` / AstroChart) и кормить её позициями из движка.
Каждый день — свежее «фото» транзитного неба, мгновенно, оффлайн.

### 3.9 Дополнительно для уюта (п.9 — мои предложения)

- **Фаза Луны** и её знак; индикатор **«Луна без курса» (VoC)** — астрологи это
  любят.
- Индикаторы **ретроградности** планет, подсветка станций.
- **«Эхо»:** «в этот день N лет назад был похожий аспект» — связать с заметками.
- **Цвето- и хапти-кодирование** по характеру аспекта (гармоничные/напряжённые):
  разный цвет и разный паттерн вибрации.
- **Темы:** глубокий «космос» (тёмная) и мягкий «рассвет» (светлая), уважать
  системную тему.
- **Бэкап/экспорт** заметок и библиотеки трактовок в JSON (и импорт) — чтобы
  ничего не потерять.
- **Поделиться днём** как картинкой (колесо + список аспектов).
- Крупные тапы, достижимость одной рукой, мягкий онбординг.
- (Nice-to-have) виджет на домашний экран Android — учесть, что через Capacitor
  это ограничено и может потребовать нативного куска; не блокирующее.

## 4. Данные и хранение

- Локальная БД: **SQLite** (Capacitor Community SQLite) или IndexedDB. Сущности:
  - `settings` (пояс, время ежедневного уведомления, орбис по умолчанию,
    состав объектов, тема);
  - `tracked_aspects` (закреплённые пары+тип);
  - `notes` (день ИЛИ сигнатура аспекта + дата + текст);
  - `interpretations` (сигнатура → текст трактовки);
  - `reminders` (на аспект: «за X дней» / «в момент»);
  - `api_key` — **только** в защищённом хранилище (не в общей БД, не в гите).
- **Сигнатура аспекта**: нормализованный ключ (planet_min, planet_max, aspect),
  чтобы «похожие» и трактовки находились надёжно.
- Экспорт/импорт всей пользовательской БД в JSON (бэкап).

## 5. Дизайн — направление для Claude Design

Передать Claude Design как **визуальное направление** (не код):

**Эстетика:** «ночное небо сквозь матовое стекло». Ультра-современный
**glassmorphism**: полупрозрачные матовые панели поверх глубокого космического
градиента, тонкий звёздный фон, мягкие свечения вокруг активных аспектов.
Всё плавно и мягко: пружинные переходы, ничего резкого.

**Палитра:** глубокий индиго/фиолет/тёмно-синий как база; светящиеся акценты —
лунное серебро, тёплое золото для гармоничных аспектов, приглушённый
розово-красный для напряжённых. Низкий контраст, но читаемо.

**Движение:** spring-анимации; свайп дня с параллаксом звёзд; хапти-тик на смене
дня; карточка аспекта «дышит» по мере приближения к точному (орбис визуально
сжимается).

**Типографика:** чистый гуманистический sans для UI; астрологические **глифы —
отдельным символьным шрифтом/SVG**; крупные читаемые числа для орбиса и времени.

**Компоненты:** матовый bottom sheet для календаря и ввода даты; чипы-фильтры
аспектов; карточка аспекта (глифы, тип, орбис, времена вход/точно/выход,
стрелка →/←); колесо неба как «герой», разворачивается на весь экран; панель
«Отслеживаю» с лентой-таймлайном.

**Микро-UX:** закреплённые аспекты — с лентой-ярлычком; пульс «приближается
точный»; тап по планете в колесе — фильтр её аспектов; долгий тап по аспекту —
добавить заметку / поставить напоминание.

### Про «рисовать картинки объектов» — что подключать

- **Глифы планет/знаков/аспектов рисовать НЕ нужно** — это символы Unicode
  (☉ ☽ ☿ ♀ ♂ ♃ ♄ ♅ ♆ ☊ ☋, знаки ♈…♓, аспекты ☌ ⚹ □ △ ☍) или бесплатный
  **астрологический символьный шрифт/SVG-набор**. Никакого генератора картинок
  для этого подключать не надо — экономия и токенов, и веса.
- **«Фото неба» — тоже без генератора:** это SVG-колесо из библиотеки (3.8).
- Генератор изображений нужен **только если** захочется декоративных
  **мифологических иллюстраций** (греческие боги для трактовок). Тогда их
  генерируют **один раз заранее** как оффлайн-ассеты (можно через Claude с
  инструментом изображений / отдельный image-генератор), кладут в приложение —
  это НЕ ежедневный расход токенов. Для старта — не обязательно.

## 6. Уведомления и фон — честные ограничения

- «Вибрация ровно в момент аспекта» при закрытом приложении достигается
  **предварительным планированием** локальных уведомлений на точные времена
  (их вычисляет движок). Постоянный фоновый процесс не нужен и на Android
  ненадёжен — не делать.
- Пересчитывать и перепланировать пачку ближайших уведомлений при запуске и
  раз в сутки (ограничение числа запланированных уведомлений ОС — учесть,
  планировать «окно» на N ближайших дней).
- Tap→событие: класть в уведомление `extra` (тип события, дата, сигнатура) и
  роутить на нужный экран.

## 7. Вехи (порядок работ)

1. **Спайк движка** + сверка с эталоном → выбор B1/B2 (go/no-go).
2. Модель данных + **лента дней** + **карточка аспекта** (оффлайн-ядро).
3. **Колесо неба** (3.8).
4. **Отслеживание / поиск / диапазон дат** (3.3).
5. **Заметки** + «похожие прошлые» (3.5).
6. **Уведомления** + deep-link на событие (3.4).
7. **Библиотека трактовок** + **чат на ключе** (3.6, 3.7).

## 9. Как пользоваться эталоном (`engine_reference/`)

Это уже работающий код на `pyswisseph`, проверенный в этом проекте. Назначение —
**эталон для сверки** нового движка (B1/B2): любой результат на устройстве
обязан совпадать с этим кодом.

- `daily_aspects_astro.py` — расчёт аспектов на день, точное время, applying/
  separating, аудит элонгаций, список объектов и аспектов.
- `daily_summary.py` — форматирование дневной сводки (образец состава полей).
- `transits_astro.py` — соединения/аспекты в диапазоне, **окно орбиса**
  (вход→точно→выход) с обработкой ретроградного качания, натальная функция и
  система домов (на будущее, если понадобятся дома).
- `transits_files.py` — генерация `.ics`/`.txt` (образец, если захочется
  экспортировать события в календарь).

**Процедура сверки (обязательна на шаге 1):** взять 5–10 контрольных дат,
посчитать позиции и аспекты питон-эталоном (он требует `pysweph`/`pyswisseph`
в окружении 3.12) и кандидат-движком на устройстве; расхождение долгот — до
0.01°, времён точных аспектов — до минуты. Зафиксировать в `CLAUDE.md`.

### 10.2 Платформа/сборка: APK на Kotlin (⚠ согласовать с разделом 1)

- Пожелание владелицы: **нативная сборка APK на Kotlin** (Android Studio).
- **Важно:** это расходится с рекомендацией раздела 1 (web + Capacitor) и ближе

  строить до этого. При любом исходе UI остаётся движок-независимым (раздел 1).

### 10.3 Проверка обновлений и номер версии при запуске (доп. к 7)

- При **каждом запуске** приложение проверяет наличие обновления.
- **Номер версии вшит** в сборку и виден в интерфейсе (экран «О приложении» /
  подвал настроек). Версия показывается всегда, в том числе оффлайн.

### 10.4 Жёсткое (принудительное) обновление от сервера (доп. к 10.3)

- Сервер может объявить **обязательное обновление**: если установленная версия
  ниже минимально допустимой — показать блокирующий экран «нужно обновиться» и
  не пускать в основной функционал, пока не обновятся.
- Проверка версии — лёгкий сетевой запрос (только номер/манифест). **Ядро
  расчётов остаётся оффлайн** (раздел 0, п.3): сеть нужна только для этой
  проверки и для чата трактовок (3.6).

### 10.5 Проверка вибрации с понятной подсказкой (доп. к 3.4, 3.9)

- При запуске (или в настройках) проверять, **доступна ли вибрация/хаптик** на
  устройстве (Capacitor Haptics / Vibration API — наличие мотора, `canVibrate`).
- Если вибрации нет или она отключена — показать **понятную инструкцию, как её
  включить** (системные настройки звука/вибрации, режим без звука и т.п.),
  пошагово под Android. Не падать молча — объяснять пользователю.

### 10.6 Локальное хранение в выбираемой папке Android (доп. к 4)

- Пользовательские данные (заметки, библиотека трактовок, бэкапы) хранятся
  **локально в папке, которую выбирает астролог** на устройстве.
- Реализация: системный выбор каталога Android (**Storage Access Framework** /
  Capacitor Filesystem с выбором папки). Выбранный путь запоминать в `settings`.
- Совместимо с экспортом/импортом JSON (раздел 4) и с принципом оффлайн-первого.

### 10.7 Импорт записей: объединение без смешивания (доп. к 3.5, 4)

- Возможность **импортировать другой набор записей** (заметки, трактовки) и
  **объединить со своими, НЕ смешивая**: импортированные записи остаются
  отличимыми от собственных, не перезаписывают их и не сливаются в общую кашу.
- В модели данных — поле **`source`** у заметок/трактовок (своё / импортировано,
  с меткой набора). Импорт — **слияние без конфликтов**: дубликаты не затирать,
  давать выбор/пометку. Фильтр по источнику в списках и поиске.

### 10.8 Облачка-подсказки при первом запуске (доп. к 3.9)

- При **первом включении** — мягкий онбординг: рядом с каждым пунктом интерфейса
  всплывают **маленькие «облачка»-пояснения** (coach marks / tooltips): что это
